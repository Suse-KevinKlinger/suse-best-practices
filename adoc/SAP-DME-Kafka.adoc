// tag::intro[]
To configure Kafka for {dme}, we use the *Strimzi Kafka Operator*, which is available in the {rac}. This operator natively handles the lifecycle, security, and scaling of the Kafka cluster on {rke}.

More details on configuring and connecting to Strimzi Kafka to {dme} can be found at the following link:
https://help.sap.com/docs/sap-digital-manufacturing/setup-and-operations-guide-for-sap-digital-manufacturing-for-edge-computing/appendix-4-set-up-and-connect-to-strimzi-kafka
// end::intro[]

// tag::prerequisites[]
==== Prerequisites

Before deploying the Strimzi Operator, you need to create the related namespace and *imagePullSecret*.
To create the namespace, run:

[source, bash]
----
kubectl create namespace kafka
----

Instructions on how to create the *imagePullSecret* can be found in <<imagePullSecret>>.

Before you can install the application, you need to log in to the registry. You can find the instructions in <<LoginApplicationCollection>>.

// tag::install-operator[]
==== Deploying the Strimzi Operator

Install the Strimzi Cluster Operator using Helm from the {rac}:

[source, bash]
----
helm install strimzi-cluster-operator oci://dp.apps.rancher.io/charts/strimzi-kafka-operator \
  --namespace kafka \
  --set replicas=3
----

// tag::deploy-cluster[]
==== Deploying the Kafka Cluster

Once the Strimzi Operator pods are running, you can deploy a Kafka cluster. To do so, create a file named `strimzi-kafka.yaml` with the following configuration. Ensure you replace the `class: <YOUR_STORAGE_CLASS>` placeholder with your actual storage class (e.g., `harvester` or `longhorn`):

[source, yaml]
----
apiVersion: kafka.strimzi.io/v1
kind: KafkaNodePool
metadata:
  name: controller
  namespace: kafka
  labels:
    strimzi.io/cluster: kafka-cluster
spec:
  replicas: 3
  roles:
    - controller
  storage:
    type: jbod
    volumes:
      - id: 0
        type: persistent-claim
        size: 5Gi
        kraftMetadata: shared
        deleteClaim: false
        class: <YOUR_STORAGE_CLASS> # e.g., harvester or longhorn
---
apiVersion: kafka.strimzi.io/v1
kind: KafkaNodePool
metadata:
  name: broker
  namespace: kafka
  labels:
    strimzi.io/cluster: kafka-cluster
spec:
  replicas: 3
  roles:
    - broker
  storage:
    type: jbod
    volumes:
      - id: 0
        type: persistent-claim
        size: 128Gi
        deleteClaim: false
        class: <YOUR_STORAGE_CLASS> # e.g., harvester or longhorn
---
apiVersion: kafka.strimzi.io/v1
kind: Kafka
metadata:
  name: kafka-cluster
  namespace: kafka
spec:
  kafka:
    version: "4.3.0"
    metadataVersion: "4.3"
    authorization:
      type: simple
    listeners:
      - name: plain
        port: 9092
        type: internal
        tls: false
        authentication:
          type: scram-sha-512
      - name: tls
        port: 9093
        type: internal
        tls: true
        authentication:
          type: scram-sha-512
    config:
      offsets.topic.replication.factor: 3
      transaction.state.log.replication.factor: 3
      transaction.state.log.min.isr: 2
      default.replication.factor: 3
      min.insync.replicas: 2
  entityOperator:
    topicOperator: {}
    userOperator: {}
----

NOTE::
The configuration above creates a Kafka cluster with 3 brokers and 3 controllers, each with persistent storage.
You should adjust the sizes according to your environment.

Apply the configuration to the cluster:

[source, bash]
----
kubectl apply -f strimzi-kafka.yaml
----

// tag::create-user[]
[#createKafkaUser]
==== Creating the Kafka User and ACLs

After the Kafka cluster is deployed, you need to create a Kafka user for {dme} and configure the necessary Access Control Lists (ACLs) to allow {dme} to read and write to its internal topics.

Create a file named `strimzi-kafka-user.yaml` with the following content:

[source, yaml]
----
apiVersion: kafka.strimzi.io/v1
kind: KafkaUser
metadata:
  name: kafka-user
  namespace: kafka
  labels:
    strimzi.io/cluster: kafka-cluster
spec:
  authentication:
    type: scram-sha-512
  authorization:
    type: simple
    acls:
      - resource:
          type: topic
          name: com.sap.dsc.dm.
          patternType: prefix
        operations:
          - Read
          - Write
          - Create
          - Describe
          - Delete
      # Group permissions
      - resource:
          type: group
          name: "*"
          patternType: literal
        operations:
          - Read
          - Describe
----

Apply the user configuration:

[source, bash]
----
kubectl apply -f strimzi-kafka-user.yaml
----

The Strimzi operator will automatically generate a Kubernetes Secret containing the user's credentials.
Retrieve and decode the password, as you will need it for the {dme} installation (<<configureKafkaDME>>):

[source, bash]
----
kubectl get secret kafka-user -n kafka -o jsonpath='{.data.password}' | base64 -d
----
// end::create-user[]

// tag::configure-dme[]
[#configureKafkaDME]
==== Connecting SAP {dme} to Strimzi Kafka

Once the Kafka cluster and user are ready, you must configure the SAP {dme} Helm `values.yaml` to disable the embedded Kafka and point to the external Strimzi deployment.

Update your `values.yaml` file with the connection details and the decoded password retrieved in the previous step (<<createKafkaUser>>):

[source, yaml]
----
global:
  kafka:
    useEmbedded: false
    auth:
      brokers: 'kafka-cluster-kafka-bootstrap.kafka.svc.cluster.local:9092'
      username: 'kafka-user'
      password: '<YOUR_DECODED_KAFKA_PASSWORD>'
----
// end::configure-dme[]
